Chaser Simulation Lifecycle

Frame-by-frame path from the simulation harness into the chaser IDAE model, through observation, memory, patterns, projections, action selection, action application, and post-action feedback.

Current code: one IDAE cycle before each applied action

  • resolves scenario, map, actors, settings
  • constructs chaserIdae

2. A frame starts

  • freezes synchronized actor positions
→
3. Simulation enters chaser IDAE
  • updateChaserReasoningState
  • stepChaserIdae
  • passes frame context into actor adapter
→
4. Generic IDAE order runs
  • stepActorIdae
  • stepDecisionEngine
  • observe, memory, patterns, projections, action, snapshot

  • getProgrammaticChaserInput
  • planProgrammaticChaserAction
→
6. Simulation commits the result
  • applyChaserAction
  • recordChaserSuccessMetrics
  • updates trace, metrics, and lastStep

One Frame In Code

stepChaseSimulation(state) -> buildReasonedActionFrame(state) -> createSynchronizedFrameContext(state) -> updateChaserReasoningState(state, frameContext) -> stepChaserIdae(state.chaserIdae, frameContext) -> stepActorIdae(...) -> stepDecisionEngine(...) -> observe -> updateMemory -> updatePatterns -> updateProjections -> chooseAction -> getSnapshot -> returns chaserReasoning.action -> commitReasonedActionFrame(state, actionFrame) -> applyChaserAction(state, actionFrame.chaserAction) -> recordChaserSuccessMetrics(state.chaserIdae, outcome) -> recordChaseTraceFrame(state.traceRecorder, state)

Simulation Access Points

Create chaser model
simulation/simulation.mjs calls createChaserIdae({ scenario }) during createChaseSimulationState.
Step chaser model
updateChaserReasoningState calls stepChaserIdae with a synchronized frame context before either actor moves.
Apply chaser output
commitReasonedActionFrame passes chaserAction into applyChaserAction, which uses applyVehicleAction and obstacle collision resolution.
Record feedback
After movement, recordChaserSuccessMetrics writes outcome memory for later frames. Run metrics and trace frames are updated in the same commit phase.

Frame Context

The chaser receives simulator truth as frame context, but the decision model only reasons through observed-world and memory shapes after the observation adapter runs.

{ chaserPosition, evaderPosition, chaserLookDirection, evaderExists, frameIndex, fieldOfViewAngleRadians, obstacles, columns, rows, projectionSettings, humanInput, programmaticChaserEnabled }

Chaser IDAE Stage Mapping

Assembly
actors/chaser/chaser-decision-model-adapter.ts calls createActorIdae and provides chaser-specific stage callbacks and module lists.
Observe
observeChaserIdae calls observeChaserEnvironment, which delegates to perception/chaser/observe.ts. This computes visible evader facts and visible map facts.
Memory
CHASER_MEMORY_MODULES runs updateChaserMemoryStage. It updates direct evader-location memory, persistent map-shape memory, map recency, and observed evader motion.
Patterns
CHASER_PATTERN_MODULES runs updateChaserPatternStage. It updates continuance and wall-avoidance pattern evidence using remembered obstacles, not full map truth.
Projections
CHASER_PROJECTION_MODULES runs updateChaserProjectionStage. It updates the evader-motion projection from pattern units, current memory, and remembered map geometry.
Action
chooseChaserIdaeAction either normalizes human input or calls getProgrammaticChaserInput, which calls planProgrammaticChaserAction in decision-model/actions/chaser/plan.ts.
Snapshot
getChaserIdaeSnapshot combines chaser knowledge with framework fields through buildActorSnapshot. UI, debug overlays, tests, and trace recording read this output.

Observed World Boundary

Perception is outside the decision model. It turns simulator truth into a trusted observed-world payload before memory or patterns see it.

observeChaserWorld(context) -> getObservedActor(...) -> getObservedMap(...) -> getVisibleMapWalls(...) -> getVisibleMapAreaCells(...)

Action Boundary

The IDAE action stage returns vehicle controls plus debug payload. The simulator owns the physical application of those controls.

chaserAction = { source: "idae", forward, reverse, steering, actionPath, selectedActionProposalId, actionProposals }

Feedback Boundary

Success metrics are recorded only after the world state changes, so they describe the committed outcome and are available to future IDAE cycles.

commitReasonedActionFrame -> applyChaserAction -> validatePredictionPerformance -> recordChaserSuccessMetrics -> updateRunMetrics

Where Debug Views Read From

Latest action
state.lastStep.chaserAction stores selected controls and action proposal debug data.
Latest reasoning
state.lastStep.chaserReasoning stores the last IDAE cycle with action and snapshot.
Decision state
state.chaserIdae.state persists memory, pattern modules, projections, engines, and controller state.
Trace/debug
debug/debug-snapshot.mjs, debug/decision-debug.mjs, and scene path rendering read these payloads.

Important Timing Detail

Chaser and evader reasoning both use the same frozen frame context before actions are applied. The chaser does not observe the evader's just-computed movement until the next simulation frame.

Prediction performance is recorded before movement and validated after the evader movement is applied.
