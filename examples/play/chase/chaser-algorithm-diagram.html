1. Runtime Tick

runtime.mjs orchestrates one frame.

R World state

  • True chaser position and heading.
  • True target position and heading.
  • Obstacles from getFieldObstacleLayout.
  • Separate actual speeds: chaser speed and target speed.

F Field of view check

  • getChaserTargetPerception computes bearing and distance.
  • Visible means inside FOV range and angle.
  • Visibility is blocked by obstacle line of sight.
  • Very close targets are treated as visible.

B Branch

Human chaser off Use keyboard input from I, A, and D.
Programmatic chaser on Use the chaser knowledge and controller pipeline below.

2. Chaser Knowledge

The chaser does not receive true target speed.

E Target motion estimate

  • createTargetMotionEstimate stores estimated position, direction, turn rate, and speed.
  • Initial speed estimate comes from DEFAULT_TARGET_SPEED_ESTIMATE_UNITS_PER_SECOND.
  • Actual target speed only moves the real target, not the estimate.

V If target is visible

  • Position is reconstructed from chaser position, heading, bearing, and distance.
  • Observed delta updates target direction.
  • Observed turn rate is derived from direction change.
  • Observed speed updates the speed estimate using an EWMA.

U If target is unseen

  • Estimate advances by estimated direction and estimated speed.
  • Estimated position is collision-resolved against known walls.
  • secondsSinceObservation accumulates for stale invalidation.

W Wall evidence

  • updateWallAvoidanceEvidence only counts when the target is visible.
  • It tracks whether observed near-wall episodes avoid or hit.
  • Prediction wall-avoidance is gated until enough episodes exist.

3. Prediction Plan

Projection is derived from chaser knowledge.

P Build prediction

  • buildTargetPredictionPlan calls predictTargetMotionWithKuramoto.
  • Signals include current direction, observed directions, turn bias, and learned wall avoidance.
  • The Kuramoto engine stays generic: it only combines directional oscillators.

X Project target path

  • buildTargetProjectionPath samples future target positions.
  • Projection speed is the chaser's estimated target speed.
  • Each projected step resolves collisions against known walls and obstacles.

I Invalidate stale plans

  • If target is unseen too long, the plan becomes non-actionable.
  • Visible target contact immediately rebuilds a fresh projection.
  • Old queued validations are cleared when visible observations diverge.

D Debug projection

  • The left-pane projection toggle only controls drawing.
  • The gameplay prediction plan exists even when projection debug is hidden.

4. Controller Output

The controller still emits only forward and steering.

C Select pursuit point

  • Ignore the projection if the plan is non-actionable.
  • Choose the first projected sample whose time is reachable by chaser speed.
  • If none is reachable, use the farthest projected sample as lookahead.
  • If no path exists, fall back to current target estimate.

G Choose goal direction

  • Priority 1: direction to pursuit point.
  • Priority 2: visible bearing direction.
  • Priority 3: search direction based on previous steering.

K Local movement consensus

  • planLocalMovementDirection creates movement signals.
  • Signals: goal, avoid-wall, follow-wall, unstick-wall.
  • Kuramoto consensus combines the active movement signals.

S Hard safety filter

  • constrainDirectionByLocalWalls runs after consensus.
  • It prevents the consensus direction from driving deeper into a wall.
  • If needed, it turns the direction into a wall-following tangent plus outward bias.

O Vehicle input

  • Desired direction is compared with current chaser heading.
  • Bearing outside deadzone returns steering -1 or 1.
  • Bearing inside deadzone returns steering 0.
  • Programmatic chaser always returns forward: true.

Knowledge boundary

The chaser's algorithm is derived from what the chaser can know: FOV perception, estimated target motion, estimated target speed, learned wall-avoidance evidence, known world geometry, and its own speed. The true target speed is only used by the world simulation to move the red target.

true target speed moves red target observed through FOV estimated target speed drives projection

Slower-chaser cutoff test

With chaser speed below target speed, the controller does not chase the current target point directly. It scans the projected samples and asks which sample is reachable by chaser speed at that sample's secondsAhead. That reachable sample becomes the cutoff goal. If no projected sample is reachable, the controller still aims at the farthest lookahead sample rather than the stale current position.

projection path sample loop distance / chaser speed <= sample secondsAhead select cutoff

Per-frame summary

runtime tick -> FOV perception -> update chaser knowledge -> build target prediction plan -> select pursuit point -> movement consensus -> hard wall safety -> steering input -> physics collision resolution
